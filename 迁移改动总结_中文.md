## 2. 迁移目标（从 ROS1 到 ROS2 Humble）

原目标不是“拷贝文件”，而是完成以下落地能力：

1. ROS1 代码在 Jetson Orin NX + Ubuntu 22.04 + ROS2 Humble 可编译、可启动。
2. 关键链路可分层排障（感知、控制、中间策略、底盘串口）。
3. 迁移后接口（topic / 参数 / message）可被团队明确复用。
4. 所有删除和替换动作有备份路径，支持回滚。

## 3. 文件级改动总览（改了什么 + 为什么改）

### 3.1 `velpub` 包（巡线主链路）

#### 3.1.1 `src/velpub/scripts/detect_line2.py`

改动点：

1. ROS1 风格改为 ROS2 节点（`rclpy`）。
2. 改成网络桥接模式：
   - Jetson 侧采集摄像头帧
   - TCP 发给电脑端推理
   - 收到 `float32` 偏差后发布
3. 统一输出：
   - `/lane/steering_angle`（核心转向偏差）
   - 可选 `/cmd_vel`（调试/兼容）
   - `note.txt`（给 `vth2ros.py` 读取）
4. 参数化：
   - `laptop_ip`、`port`、`video_device`、`loop_hz`、`reconnect_interval_sec`
5. 稳定性增强：
   - 摄像头打开策略：优先 `V4L2`
   - 打开后立刻读帧校验，防止“假打开”
   - 重连与错误日志节流
   - 退出时保护发布，避免 context invalid

为什么改：

- ROS1 `rospy` 版与 Humble 不兼容。
- Jetson 端不适合做本地窗口推理显示，必须把推理移到电脑端并保留 ROS2 接口。
- 原先大量“启动了但无输出/假输出”的问题本质是摄像头与网络边界不清，参数化后可快速定位。

#### 3.1.2 `src/velpub/launch/detect_line.launch.py`

改动点：

1. ROS2 launch 化。
2. 补齐并暴露关键参数（IP、端口、视频设备、分辨率、jpeg 质量、重连周期、note 文件路径）。

为什么改：

- 现场设备经常变化（电脑 IP、摄像头编号），写死会导致每次改代码。
- launch 参数化可以直接命令行覆盖，降低调试成本。

#### 3.1.3 `src/velpub/scripts/vth2ros.py`

改动点：

1. ROS2 节点化。
2. 定时读取 `note.txt` 的偏差值。
3. PID 输出 `Twist` 到 `/cmd_vel_1`。
4. PID 与频率参数化（`P/I/D`、`DeadArea`、`publish_hz`、`Max_angular_curve`）。

为什么改：

- 保留原“偏差 -> 速度控制”算法思路，同时满足 ROS2 执行链路。

#### 3.1.4 `src/velpub/launch/AIcar_move.launch.py`

改动点：

1. ROS2 启动入口。
2. 暴露 PID 与话题参数。

为什么改：

- 统一到 ROS2 启动体系，便于多终端按固定顺序起服务。

#### 3.1.5 `src/velpub/scripts/velpub.py`

改动点：

1. ROS2 化交通标志状态机节点。
2. 订阅 `/cmd_vel_1`，处理后发布 `/cmd_vel`。
3. 继续读取外部标志识别文件（`internal_memory.txt`）用于策略分支。

为什么改：

- 该节点是“策略融合层”，不是底盘驱动，必须保留。
- 便于将纯巡线控制与交通规则控制解耦。

#### 3.1.6 `src/velpub/launch/get_yolo_rst.launch.py`

改动点：

1. ROS2 launch 化，仅启动 `velpub.py`。

为什么改：

- 保持与原三段式启动一致（detect_line -> AIcar_move -> get_yolo_rst）。

#### 3.1.7 `src/velpub/CMakeLists.txt`、`src/velpub/package.xml`

改动点：

1. ROS2 安装声明（`install(PROGRAMS ...)`）补齐，确保脚本进入 `install/velpub/lib/velpub`。
2. 依赖声明补齐（`rclpy`、`geometry_msgs`、`std_msgs`、`launch_ros`、`python3-opencv`、`python3-numpy`）。

为什么改：

- 解决 `ros2 launch ... executable not found`。
- 解决运行时依赖缺失。

### 3.2 `turn_on_wheeltec_robot`（底盘驱动）

#### 3.2.5 `src/turn_on_wheeltec_robot/CMakeLists.txt`、`src/turn_on_wheeltec_robot/package.xml`

## 4. 启动链路（ROS2 对照）

建议四终端固定顺序：

1. 底盘串口：
   - `ros2 launch turn_on_wheeltec_robot turn_on_wheeltec_robot.launch.py usart_port_name:=...`
2. 巡线桥接：
   - `ros2 launch velpub detect_line.launch.py laptop_ip:=... port:=... video_device:=...`
3. 偏差转速度：
   - `ros2 launch velpub AIcar_move.launch.py`
4. 规则融合：
   - `ros2 launch velpub get_yolo_rst.launch.py`

链路为：

`PC推理服务 -> detect_line2.py -> note.txt -> vth2ros.py -> /cmd_vel_1 -> velpub.py -> /cmd_vel -> wheeltec_robot`

## 5. 重点复盘：为什么“代码都启动了，小车还是不跑”

本次排障中，出现过“看起来都在运行，但车不动”的典型情况。  
根因不是单一问题，而是多因素叠加。

### 5.1 现象 1：`/cmd_vel_1` 和 `/cmd_vel` 有频率，但车不动

日志/现象：

1. `ros2 topic hz /cmd_vel_1`、`ros2 topic hz /cmd_vel` 显示约 30Hz。
2. 但 `ros2 node list | grep wheeltec_robot` 查不到底盘节点。

结论：

- 上层控制链路在跑，但底盘驱动没跑，命令无人执行。

### 5.2 现象 2：底盘节点启动报错

报错：

- `Failed to open serial port /dev/wheeltec_controller: No such file or directory`
- `Permission denied`

结论：

1. 默认串口别名 `/dev/wheeltec_controller` 未建立（udev 未生效或规则不匹配）。
2. 用户不在 `dialout` 组的生效会话中（`usermod` 后未重登）。

### 5.3 现象 3：串口已打开，仍可能不动

说明：

1. `serial port opened` 仅说明设备文件可打开。
2. 如果串口不是底盘主控口，或被其他节点覆盖 `/cmd_vel`，仍可能不动。

排查重点：

1. `/odom`、`/imu1` 是否持续有数据（确认主控协议对上）。
2. `/cmd_vel` 发布者是否唯一且合理（`ros2 topic info /cmd_vel -v`）。
3. 手动发布较明显速度（例如 `0.20`）是否能触发动作。

### 5.4 现象 4：巡线节点启动了但无有效输入

问题链：

1. 摄像头设备占用（`Device '/dev/video0' is busy`）。
2. 非显示环境 `cv2.imshow` 报 GTK 错误。
3. 摄像头后端“假打开”，实际 `read()` 一直失败。

结论：

- 上层 `detect_line` 看似启动，实际没有稳定偏差输出，后续速度链路输入异常。

### 5.5 现象 5：Python 环境污染导致运行失败

问题：

1. `numpy`、`Pillow`、`torchvision` 来自用户目录和系统目录混用。
2. 出现 `module ... has no attribute ...`。

结论：

- ROS2 节点运行解释器与依赖不一致，导致节点直接崩溃或异常。

## 6. 已验证有效的修正方案（本轮）

1. 底盘串口：
   - 显式指定真实设备（如 `/dev/ttyUSB1` 或 `/dev/serial/by-id/...`）。
   - 让 `dialout` 权限生效（重登或 `newgrp dialout`）。
2. 摄像头：
   - 仅使用 V4L2 打开策略。
   - 打开后先读帧验证，不通过即重试。
3. 控制链：
   - 明确四节点启动顺序，分层验证每一段。
4. Python 依赖：
   - 清理用户态污染包，回归系统依赖。

## 7. 当前遗留项（必须进入后续计划）

1. `interfaces` 下 action/srv 在部分副本仍为空文件，需要按业务接口补齐正式字段。
2. `turn_on_wheeltec_robot` 默认 `usart_port_name` 建议改成稳定 by-id 名称，减少现场参数传递。
3. 多个 `package.xml` 仍是迁移占位邮箱（如 `codex@local`），需统一为项目维护邮箱。
4. 建议固化 udev 规则与串口映射（把 `/dev/wheeltec_controller` 恢复为稳定可用别名）。

## 8. 对项目组的结论

本次迁移做的不是“机械换语法”，而是完成了完整工程落地：

1. ROS1 到 ROS2 的接口迁移与可构建化。
2. 感知、控制、策略、底盘分层解耦，支持分段排障。
3. 现场关键参数（摄像头、串口、网络）全部参数化。
4. 对“启动了但车不动”给出了明确的根因路径和可执行修复步骤。

这使得项目可以在 Jetson Orin NX + Ubuntu 22.04 + ROS2 Humble 下持续联调，而不是一次性演示。
