# YOLOv8 长期迁移计划书

## 1. 文档目的

本文用于给项目组评审和对齐 `yahboom_ws` 小车视觉识别链路的长期演进方向。

当前建议的长期路线不是继续在旧版 `YOLOv5 cfg/wts + DeepStream-Yolo native builder + internal_memory.txt` 这条链上做深度维护，而是逐步迁移到：

```text
YOLOv8
-> ONNX / TensorRT
-> 新检测推理链
-> 标准化输出接口
-> 兼容现有规则控制层
-> 最终替换旧文件接口
```

本文强调的是“分阶段迁移、不中断现有可跑链路、控制风险”的工程方案。

## 2. 当前系统现状

### 2.1 当前真正跑车的主链

当前小车主链以 `velpub` 为核心，实际控制链为：

```text
detect_line3.py
-> note.txt
-> vth2ros.py
-> /cmd_vel 或 /cmd_vel_1
-> velpub.py
-> /cmd_vel
-> wheeltec_robot_node
-> 底盘
```

其中：

- `detect_line3.py` 负责本地巡线推理与网页视频流
- `vth2ros.py` 负责把 `deviation,severity` 转成巡线速度
- `velpub.py` 负责基于交通标志识别结果进行规则仲裁
- `wheeltec_robot_node` 负责底盘串口下发

### 2.2 当前 YOLO 的角色

当前 YOLO 并不是直接通过标准 ROS 话题进入主控制链，而是通过 DeepStream-Yolo 的自定义输出写入：

```text
/opt/nvidia/deepstream/deepstream-6.0/sources/DeepStream-Yolo/nvdsinfer_custom_impl_Yolo/internal_memory.txt
```

`velpub.py` 读取该文件中的识别结果，再进入交通规则状态机。

这说明当前系统存在两个明显特点：

- 视觉识别链与控制链之间通过“文件接口”耦合
- YOLO 部署方式与当前 Jetson / DeepStream / TensorRT 版本存在较强历史包袱

### 2.3 当前主要问题

当前系统的长期问题主要包括：

1. 旧版 `YOLOv5 cfg/wts` 构图方式在 `DeepStream 7.1 / TensorRT 10` 环境下兼容性差。
2. `internal_memory.txt` 属于非标准、不可类型检查、不可版本约束的接口。
3. 规则层 `velpub.py` 与 YOLO 类别编码紧耦合，迁移成本集中在“输出兼容”而不是单纯换模型。
4. 当前链路里存在脚本默认值、launch 默认值、历史文档之间不完全一致的问题，放大了维护风险。

## 3. 迁移目标

本项目的长期目标不是单纯把模型文件从 `YOLOv5` 换成 `YOLOv8`，而是完成以下 4 件事：

1. 使用 `YOLOv8` 作为新的长期检测模型基座。
2. 在 Jetson 当前环境中建立稳定、可维护的推理部署方式。
3. 在不破坏现有跑车逻辑的前提下，让新检测链逐步接管交通规则识别。
4. 最终把 `internal_memory.txt` 替换为标准化的 ROS 接口或明确的结构化输出接口。

换句话说，目标是“模型升级 + 推理链升级 + 接口升级 + 规则层逐步兼容”。

## 4. 总体迁移策略

### 4.1 核心原则

迁移必须遵循以下原则：

- 先并行，后切换，不一步替换
- 先验证检测，再接规则，再接整车
- 先兼容旧规则接口，再考虑重构规则接口
- 巡线主链保持可独立运行，避免识别链升级影响底盘基础能力

### 4.2 总体分期

建议把整个迁移拆成 5 个阶段：

1. 需求冻结与接口基线确认
2. YOLOv8 检测链独立落地
3. YOLOv8 输出兼容现有规则层
4. 规则链切换与整车联调
5. 标准化接口替代 `internal_memory.txt`

这样做的关键好处是：

- 每个阶段都可独立验收
- 出现问题时可快速回退
- 项目组可以并行推进模型、推理部署、ROS 接口和整车联调

## 5. 分阶段详细计划

## 阶段 1：需求冻结与接口基线确认

### 目标

在开始任何迁移前，先把“当前规则层真正依赖什么”固定下来，避免后续各组对输出格式和类别含义理解不一致。

### 任务

1. 梳理 `velpub.py` 当前消费的所有标志编码与行为映射。
2. 明确每个类别的实际含义、触发条件、优先级和持续时间。
3. 固化现有 `internal_memory.txt` 的最小语义集合：
   - 至少需要什么字段
   - 哪些字段是真正被 `velpub.py` 用到的
   - 哪些字段只是历史遗留
4. 形成一份“规则层输入契约”文档。

### 产出物

- 《YOLO 规则输入契约说明》
- 《当前标志类别与控制动作映射表》
- 《现有 `velpub.py` 状态机输入依赖清单》

### 验收标准

- 项目组成员能统一回答每个标志对应什么行为
- 能明确列出哪些类别必须在 YOLOv8 中保留
- 能明确列出哪些旧类别可以废弃或延后

## 阶段 2：YOLOv8 检测链独立落地

### 目标

先让 `YOLOv8` 在 Jetson 上稳定跑起来，但此时不接管小车控制。

### 建议路线

优先建议：

```text
YOLOv8
-> 导出 ONNX
-> TensorRT / DeepStream 7.1 新链路
-> 独立检测进程
```

不建议继续沿用旧版 `cfg/wts` 构图路线。

### 任务

1. 明确 YOLOv8 训练版本、类别表和标签定义。
2. 导出 ONNX，并在 Jetson 上完成基础推理验证。
3. 验证推理速度、显存占用、加载时延和摄像头输入适配。
4. 输出独立的检测结果日志、图片或可视化结果。
5. 保证新检测链即使失败，也不影响当前巡线主链运行。

### 技术要求

- 新检测链必须可独立启动和停止
- 不依赖当前 `velpub.py` 才能验证结果
- 需要支持离线图片 / 视频 / 实时摄像头三种验证方式

### 产出物

- 可独立运行的 YOLOv8 检测程序
- ONNX 模型与部署说明
- Jetson 上的性能测试记录

### 验收标准

- 在 Jetson 上稳定启动
- 指定类别能够被正确识别
- 连续运行一段时间不崩溃
- 推理性能满足项目最低实时性要求

## 阶段 3：YOLOv8 输出兼容现有规则层

### 目标

让 YOLOv8 的输出先“兼容旧规则层”，这样可以最小成本验证新检测是否能驱动现有 `velpub.py` 状态机。

### 核心思路

不要一上来就改 `velpub.py` 的全部输入方式，而是先增加一层“适配器”。

建议增加：

```text
YOLOv8 检测输出
-> 适配器节点 / 适配器进程
-> 生成旧规则层可消费的输出
```

### 适配方式建议

优先级建议如下：

1. 短期兼容方式：
   - 继续生成 `internal_memory.txt`
   - 内容语义与当前 `velpub.py` 保持兼容
2. 中期双写方式：
   - 同时写 `internal_memory.txt`
   - 同时发布标准 ROS topic
3. 长期正式方式：
   - 规则层完全改读 ROS topic / 标准消息

### 任务

1. 定义 YOLOv8 原始检测结果到旧标志编码的映射关系。
2. 实现一个适配器，把 YOLOv8 输出转换为当前规则层可识别的数据。
3. 保证适配器输出稳定，不会因为偶发漏检导致规则层频繁抖动。
4. 增加必要的过滤、确认帧数、时序稳定策略。

### 风险点

- YOLOv8 的类别命名和旧编码未必一一对应
- 老规则层的行为触发依赖时间和连续帧，不是单帧识别就够
- 某些行为可能依赖面积、位置或距离感，不是仅靠类别标签

### 产出物

- YOLOv8 到旧规则输入的适配器
- 类别映射表
- 识别稳定性策略说明

### 验收标准

- `velpub.py` 在不改核心状态机逻辑的前提下，可以被新检测链驱动
- 主要交通规则行为可复现
- 相同场景下，新链行为与旧链基本一致

## 阶段 4：规则链切换与整车联调

### 目标

让新 YOLOv8 检测链真正进入整车控制闭环，完成从“并行验证”到“实际接管”的切换。

### 联调范围

联调时必须覆盖以下组合：

1. 巡线独立运行
2. YOLOv8 检测独立运行
3. 巡线 + 规则仲裁同时运行
4. 摄像头异常、识别异常、断流情况下的降级行为

### 任务

1. 确认启动顺序和 launch 方式。
2. 固定 `cmd_vel_1 -> velpub.py -> /cmd_vel` 这条规则仲裁链。
3. 对每种关键交通标志做实车联调。
4. 调整识别稳定阈值、状态保持时间和误触发抑制逻辑。
5. 加入必要的日志与调试开关，方便现场复盘。

### 验收场景

至少应覆盖：

- 红灯停车 / 绿灯恢复
- 左转 / 右转
- 人行道停车
- 减速区减速
- 匝道进入 / 离开
- 禁止通行绕行
- 无识别结果时正常巡线
- 误检出现时不应立即触发危险动作

### 产出物

- 新版整车启动说明
- 联调记录
- 关键场景测试视频与结果表

### 验收标准

- 新检测链接入后，小车仍能稳定巡线
- 规则行为与预期一致
- 异常场景下不会失控
- 具备可回退到旧链的能力

## 阶段 5：标准化接口替代 `internal_memory.txt`

### 目标

在新检测链稳定后，彻底摆脱 `internal_memory.txt` 这种历史文件接口，改成标准化消息接口。

### 建议方向

建议把规则层输入改成 ROS topic，例如：

- `/traffic_sign/events`
- `/traffic_sign/detections`
- `/traffic_sign/decision_input`

消息内容至少应包含：

- 类别 ID
- 类别名
- 置信度
- 框位置
- 时间戳
- 可选的稳定性状态或确认状态

### 任务

1. 定义标准 ROS message。
2. 给 YOLOv8 检测链直接发布标准消息。
3. 改造 `velpub.py`，让其优先读 ROS 消息而不是文件。
4. 保留一段时间双接口兼容，最后删除文件接口。

### 收益

- 接口可类型检查
- 可录包、可回放、可调试
- 减少文件读写竞争和时序不确定性
- 更容易扩展到其他识别模型和其他控制模块

### 验收标准

- 整车不再依赖 `internal_memory.txt`
- 所有规则输入都能从 ROS 话题观察到
- 回放 bag 时可复现规则行为

## 6. 技术架构建议

### 6.1 推荐架构

长期推荐如下结构：

```text
Camera
-> YOLOv8 Detector
-> Detection Adapter / Rule Event Publisher
-> Rule Arbiter
-> /cmd_vel
-> Base Driver
```

巡线链与规则链并存：

```text
Lane Camera / Shared Camera
-> detect_line3.py
-> note.txt or future lane topic
-> vth2ros.py
-> /cmd_vel_1

Traffic Detector
-> YOLOv8
-> rule event topic
-> velpub.py or successor rule arbiter
-> /cmd_vel
```

### 6.2 关于 `velpub.py`

建议不要在第一阶段就重写 `velpub.py` 全部状态机。

更稳妥的做法是：

- 第一阶段保留状态机
- 第二阶段做输入适配
- 第三阶段再逐步清理全局变量、旧编码和历史逻辑

也就是说，先把“输入现代化”，再做“状态机重构”。

## 7. 人员分工建议

建议按 4 个方向拆分：

### A. 模型与数据组

负责：

- YOLOv8 训练
- 类别定义
- 数据集清洗
- 模型精度验证

### B. Jetson 部署组

负责：

- ONNX / TensorRT / DeepStream 部署
- 摄像头接入
- 性能调优
- 运行稳定性

### C. 接口与中间件组

负责：

- 检测输出适配器
- ROS message 设计
- 新旧接口兼容

### D. 整车联调组

负责：

- `velpub.py` 接入验证
- 行为逻辑测试
- 实车场景验证
- 回归测试与回退策略

## 8. 风险评估

### 主要风险

1. YOLOv8 检测效果与旧类别体系不完全一致。
2. Jetson 上的性能可能不满足实时要求。
3. 新检测输出和旧规则状态机存在语义偏差。
4. 实车联调时误检会被规则层放大为错误动作。
5. 文件接口与 ROS 接口双维护期间容易出现数据不一致。

### 风险应对

- 所有阶段都保留旧链回退能力
- 先离线验证，再实车联调
- 先旁路运行，再接入控制
- 增加日志、录包和可视化工具
- 对关键行为加入连续确认和保护条件

## 9. 里程碑建议

建议按以下里程碑推进：

### 里程碑 M1

完成当前规则输入契约整理，冻结类别与行为映射。

### 里程碑 M2

YOLOv8 在 Jetson 上完成独立推理验证。

### 里程碑 M3

YOLOv8 输出可通过适配器驱动旧规则层。

### 里程碑 M4

完成整车联调，YOLOv8 进入可用试运行阶段。

### 里程碑 M5

规则链完全摆脱 `internal_memory.txt`，切换到标准 ROS 接口。

## 10. 建议结论

对当前项目，建议的总体决策如下：

1. 长期方向确定为 `YOLOv8`。
2. 不再把“硬修旧版 YOLOv5 cfg/wts 部署链”作为长期主路线。
3. 迁移时先保留现有规则层，优先做输出兼容，而不是一次性推倒重来。
4. 在整车级别，必须坚持“并行验证 -> 小范围切换 -> 全量替代”的迁移节奏。

最终建议是：

```text
短期：维持现有可跑链路
中期：YOLOv8 并行验证并接入旧规则层
长期：用标准 ROS 接口替代 internal_memory.txt，完成整套识别链现代化
```

## 11. 下一步建议

如果项目组认可本计划，建议下一步立即启动以下三件事：

1. 输出《当前标志类别与控制动作映射表》
2. 确认 YOLOv8 目标类别清单
3. 确定新检测链第一版采用 ONNX + TensorRT 还是 DeepStream 7.1 新管线

这三件事一旦明确，项目就可以进入正式实施阶段。
