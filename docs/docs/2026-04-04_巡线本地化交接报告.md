## 3. 当前启动链

### 3.1 底盘

底盘 launch 默认已经写死到：

- 串口：`/dev/ttyUSB0`
- 波特率：`115200`

对应文件：

- `F:\robot\test\jetson\yahboom_ws\src\turn_on_wheeltec_robot\launch\base_serial.launch.py`


### 3.2 纯巡线推荐链路

当前推荐的“纯巡线调试”链路：

```bash
ros2 launch turn_on_wheeltec_robot turn_on_wheeltec_robot.launch.py
ros2 launch velpub detect_line.launch.py
ros2 launch velpub AIcar_move.launch.py cmd_vel_topic:=/cmd_vel
```

注意：

1. 纯巡线调试时，不要启动：

```bash
ros2 launch velpub get_yolo_rst.launch.py
```

2. 因为 `get_yolo_rst.launch.py` 会启动 `velpub.py`，它会把巡线输出当成“低优先级输入”，只在 `normal pub` 分支下放行。

### 3.3 Yolo + 巡线仲裁链路

如果后续需要恢复带规则仲裁的链路：

```bash

ros2 launch velpub detect_line.launch.py
ros2 launch velpub AIcar_move.launch.py cmd_vel_topic:=/cmd_vel_1
ros2 launch velpub get_yolo_rst.launch.py
```

这时：

- `/cmd_vel_1`：巡线原始输出
- `/cmd_vel`：`velpub.py` 仲裁后的最终输出

## 4. detect_line2 与 detect_line3 的区别

### 4.1 detect_line2.py

职责：

1. Jetson 采图
2. JPEG 编码
3. 发给电脑端 TCP 服务
4. 收回 `deviation,severity`
5. 写 `note.txt`
6. 发布：
   - `/lane/steering_angle`
   - `/lane/turn_severity`

适用场景：

- 继续使用电脑端推理服务
- 需要复用原来 Windows 侧的算法服务端

### 4.2 detect_line3.py

职责：

1. Jetson 本地采图
2. Jetson 本地加载 UNet 权重
3. 本地计算 `deviation,severity`
4. 本地写 `note.txt`
5. 本地发布：
   - `/lane/steering_angle`
   - `/lane/turn_severity`
6. Jetson 本地启动一个轻量 HTTP/MJPEG 服务，提供三路视频：
   - `Original`
   - `Mask`
   - `Fit`

适用场景：

- 不想依赖电脑在线
- 想降低网络往返延迟
- 想让电脑断开后小车继续跑

### 4.3 detect_line3.py 的切换方式

没有新增 launch 文件。

当前仍然复用：

- `F:\robot\test\jetson\yahboom_ws\src\velpub\launch\detect_line.launch.py`

要切换脚本，只需要改其中的：

```python
executable="detect_line3.py"
```

如果要切回旧模式，就改回：

```python
executable="detect_line2.py"
```

## 5. lane_local_core.py 是什么

`lane_local_core.py` 是从：

- `F:\robot\transport_test\car\4.4server(python).py`

里抽出来的“纯计算核心”，主要保留了：

1. `LaneTracker`
2. `load_unet_model`
3. `run_unet_inference`
4. `postprocess_lane_mask`
5. `build_debug_visualization`

也就是说：

- `lane_local_core.py` 负责算法
- `detect_line3.py` 负责 Jetson 上的摄像头、ROS、网页流服务

这是一次刻意的职责分离，便于后续维护。

## 6. 本地化后的实际收益

本地化之后，已经确认的收益是：

1. 少了 Jetson -> 电脑 -> Jetson 的往返延迟。
2. 纯巡线响应更直接，主观观感明显比旧模式顺。
3. 看到转弯线时，响应比以前更快。
4. 电脑断开时，只影响看图，不影响小车继续跑。
5. 电脑重新访问网页即可看到新的实时画面。

当前项目内最有价值的结论是：

- 本地模式值得继续投入，不建议再把主要巡线调参精力放回旧的远端推理模式。

## 7. detect_line3.py 当前行为

### 7.1 视频流

`detect_line3.py` 启动后，会在 Jetson 本地启动网页流服务。

默认端口：

- `8090`

浏览器可访问：

```text
http://<jetson-ip>:8090/
```

页面里有三路图：

1. `Original`
2. `Mask`
3. `Fit`

### 7.2 自动打印访问地址

`detect_line3.py` 已经加入“自动探测 Jetson 当前 IPv4 并打印可访问地址”的逻辑。

预期启动日志会输出类似：

```text
detect_line3 started: video=/dev/video3, stream=http://192.168.1.xx:8090/
```

### 7.3 电脑断开后的行为

因为视频服务是 Jetson 本地开的，所以：

1. 电脑浏览器关闭，不影响小车继续跑。
2. 电脑网络短时断开，不影响小车继续跑。
3. 电脑恢复后，重新打开网页就能接着看最新画面。

## 8. vth2ros.py 当前情况

`vth2ros.py` 现在读取的 `note.txt` 格式已经不是单个数，而是：

```text
deviation,severity
```

它会：

1. 用 `deviation` 进 PID
2. 用 `severity` 做：
   - 弯道自动降速
   - 转向自动增强

当前默认参数比老版本更适合急弯：

- `Speed_stright=0.16`
- `Max_angular_curve=2.6`
- `P=0.0200`
- `I=0.0008`
- `D=0.0030`
- `DeadArea=2.5`
- `TurnSlowdownGain=0.70`
- `TurnAngularBoost=0.40`
- `MinSpeedRatio=0.40`

这些默认值都在：

- `F:\robot\test\jetson\yahboom_ws\src\velpub\launch\AIcar_move.launch.py`

## 9. get_yolo_rst.launch.py / velpub.py 当前定位

这一层目前不要和“纯巡线调参”混在一起。

原因：

1. `velpub.py` 是仲裁器，不是纯巡线节点。
2. 它会读取 `internal_memory.txt`。
3. 它对巡线的优先级是“低于规则控制”的。
4. 只有 `normal pub` 分支下，巡线结果才会基本原样通过。

所以当前建议：

- 先把纯巡线调到稳定
- 再把 Yolo / 规则仲裁接回来

## 10. 构建与同步注意事项

### 10.1 什么时候要重建

以下情况必须重新构建：

1. 新增脚本文件
2. 改 `CMakeLists.txt`
3. 改 `package.xml`
4. 新增被 launch 直接调用的可执行脚本

当前建议使用：

```bash
cd ~/yahboom_ws
source /opt/ros/humble/setup.bash
colcon build --packages-select velpub --symlink-install
source install/setup.bash
```

### 10.2 可执行权限

Python 脚本如果被 launch 当作 `executable`，要确保有执行权限：

```bash
chmod +x ~/yahboom_ws/src/velpub/scripts/detect_line2.py
chmod +x ~/yahboom_ws/src/velpub/scripts/detect_line3.py
```

### 10.3 编码问题

`lane_local_core.py` 来自 Windows 侧脚本改编，编码和中文注释比较敏感。

这类文件如果在 Jetson 上报：

- `SyntaxError`
- `unicode error`

优先怀疑编码问题，而不是先怀疑算法逻辑。
