# 演讲稿 02：近期工作进展

## 标题

YOLOv8 迁移与规则链对接近期工作汇报

## 适用时长

约 4 到 5 分钟

## 演讲正文

大家好，我这部分主要汇报的是近期围绕小车 YOLOv8 迁移和规则链对接所完成的工作。

这部分工作的背景是，原有小车识别链依赖旧 YOLO 部署方式和历史文件接口，虽然可以跑，但长期维护成本高，而且和当前环境兼容性也存在问题。所以这段时间我们做的事情，不是简单换一个模型，而是按照“小步快跑、逐层替换”的方式，把新的 YOLOv8 检测链逐步接入现有系统。换句话说，我们不是在做一次简单的模型替换，而是在做一套从检测链、接口链到规则链的迁移。

从整体方法上讲，我们把这次迁移拆成了几个清晰层次。第一层，是把现状冻结下来。我们把当前主链、旧规则层依赖、标志类别和动作关系都整理成了文档。这样做的目的，是给后面的所有修改提供基线。后续不管是换模型、改映射，还是继续重构规则层，都不会出现“大家各自理解一套规则”的情况。

第二层，是把 YOLOv8 的新检测链独立搭起来。现在新检测节点已经可以单独启动、读取摄像头、加载模型、发布标准检测消息。也就是说，YOLOv8 不再只是一个临时实验脚本，而是已经有了正式的 ROS 节点和标准消息接口。这一点的意义在于，检测层已经独立成型，不再需要和旧规则层绑死才能运行。

第三层，是建立适配层和旧规则层的桥接。我们没有一开始就重写旧规则状态机，而是增加了一个适配层，把新检测结果转成旧规则层能够理解的输入。这样新链和旧链之间有了一个缓冲层，后续可以逐步替换，而不是一次性推倒重来。对工程来说，这种做法更稳妥，也更便于定位问题，因为我们能够清楚区分：是模型识别问题，还是映射问题，还是旧规则逻辑问题。

在这基础上，我们还做了几件关键的工程化工作。

第一，定义了新的标准消息，包括单目标检测、多目标检测以及规则输入消息。这样检测链、适配层和规则层之间不再只能通过旧文件接口沟通，而是有了标准的 ROS 消息结构。这个变化非常关键，因为它让整个链路变得可观测、可录制、可回放。

第二，把模型路径、类别映射和检测参数都做成了配置化入口。也就是说，后续接新模型时，不需要再改核心代码，只需要替换模型文件、调整 profile 和 rule map。这个改动降低了维护成本，也降低了多人协作时误改代码逻辑的风险。

第三，我们补了离线验证工具和无模型联调工具。离线工具可以录制和回放检测消息，无模型联调工具则允许在模型尚未完全准备好时，先用模拟检测结果去调试规则层。这一点很重要，因为它避免了整个迁移工作被“模型还没训练好”完全卡住。也就是说，模型训练和规则联调不再是完全串行，而是可以并行推进。

第四，我们对摄像头和显示链也做了工程化处理。摄像头设备不再依赖手工试错，而是经过逐个验证，明确哪些是可采集设备，哪些不是。并且在 YOLO 节点里加入了摄像头回退逻辑，如果默认设备打不开，会自动尝试后备设备。这样现场设备变化不会直接导致整条检测链失效。

最近几天的工作重点，则进一步往业务闭环上推进。项目组上传新的 `best.pt` 后，我们已经完成了模型接入、类别读取、中文类别映射以及网页展示增强。新的模型已经能够识别诸如“红灯”“绿灯”“左转”“右转”“人行道”“匝道”等类别，并且检测结果可以送入上层规则输入。

基于这些结果，我们已经验证了这样一条链路：YOLO 检测结果先被标准化，再通过适配层映射成旧规则层可用的 `legacy_sign_id`，最后进入 `velpub.py`。这意味着，系统已经具备了“检测结果进入动作逻辑”的前提条件。这一步其实是整个迁移中非常关键的里程碑，因为它证明新模型不是孤立运行，而是已经能够对接旧业务逻辑。

在具体规则行为上，我们最近还专门处理了行人场景。现在行人类别已经重新接回规则层，逻辑是：先识别到行人，小车立即停车；行人移开后，再恢复到进入行人场景之前的状态继续行驶。这里要强调一点，这不是简单地把某个类别映射成一个数字，而是把“识别到行人”“持续停车”“行人移开后恢复”这一整套先后关系接回到了旧规则层里。这部分的意义在于，我们已经开始从“只验证识别”走向“验证识别到动作”的闭环。

同时，这个阶段也暴露出几个真实问题。比如有些类别的业务语义还没有最终定下来，像“警告”“解除限速”“行人”等，它们到底在旧规则层里分别应该对应什么动作，还需要项目组进一步确认。再比如，有些旧逻辑本身就存在历史问题，像匝道相关的分支，目前代码本身还有待进一步复核。所以现在我们做的不是简单说“全部已经好了”，而是明确地区分：哪些部分已经打通，哪些部分还需要业务侧拍板，哪些部分是旧规则本身还需要修。

如果总结近期工作的核心成果，我认为可以概括为四点。

第一，YOLOv8 迁移的工程骨架已经完整建立，不再停留在方案阶段。  
第二，新检测链已经能够稳定输出，并通过适配层进入旧规则层。  
第三，我们已经具备了“网页端 + 消息链 + 规则链”的多层调试能力。  
第四，我们已经开始在具体业务规则上做闭环验证，说明系统正在从“能检测”走向“能驱动动作”。

目前还没有完全结束的部分，主要集中在两类：一类是部分类别的业务语义还需要项目组进一步确认，比如“警告”“解除限速”“行人”等；另一类是整车级联调还要继续推进，也就是最终把检测结果、规则逻辑和底盘动作全部放到赛道环境里完整跑通。

从工程阶段来看，当前最重要的判断是：YOLOv8 迁移已经越过了最难的“结构搭建期”，下一步的重点将转向“规则语义确认”和“整车联调收敛”。也就是说，我们已经不再主要花时间回答“这套链路能不能接起来”，而是开始回答“接起来以后能不能稳定、准确地执行目标动作”。

如果项目组后续接着推进，我建议工作的重点应该继续保持清晰分层：检测问题看检测链，映射问题看适配层，动作问题看旧规则层和 `/cmd_vel`。只有这样，后续联调才不会重新回到“问题一堆，但不知道卡在哪”的状态。

我这部分汇报完毕。

## 可选结尾一句

换句话说，现在的关键问题已经不再是“能不能接上”，而是“怎么把已经接上的链路调到足够稳定”。
