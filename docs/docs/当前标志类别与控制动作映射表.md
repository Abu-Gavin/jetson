# 当前标志类别与控制动作映射表

## 1. 文档定位

本文是 YOLOv8 迁移前必须冻结的“规则输入基线文档”。

它的用途是：

- 固定当前 `velpub.py` 真正消费的标志编码
- 固定每个编码触发的控制行为
- 明确哪些是“当前已生效逻辑”
- 明确哪些是“历史遗留/注释掉的逻辑”

后续 YOLOv8 的类别设计、适配器设计、`internal_memory.txt` 兼容策略、ROS 标准消息设计，都必须以本文为参考。

## 2. 当前规则层输入来源

当前规则层 `velpub.py` 从以下路径读取识别结果：

```text
/opt/nvidia/deepstream/deepstream-6.0/sources/DeepStream-Yolo/nvdsinfer_custom_impl_Yolo/internal_memory.txt
```

当前代码读取方式是：

- 第 1 行：`sign`
- 第 2 行：`area`

但按当前已生效逻辑看：

- `sign` 是真正驱动状态机的核心字段
- `area` 目前基本未参与有效判断

结论：

- 当前最小兼容语义核心是 `sign`
- `area` 暂时保留兼容，但不是当前第一优先级输入

## 4. 当前状态机总览

当前 `velpub.py` 的主要状态包括：

| 状态名 | 状态值 | 含义 |
| --- | --- | --- |
| `STATE_NORMAL` | `0` | 正常巡线放行 |
| `STATE_RED_LIGHT` | `1` | 红灯停车等待 |
| `STATE_GREEN_LIGHT` | `2` | 绿灯恢复过渡 |
| `STATE_LEFT_SIGN` | `3` | 检测到左转标志后的等待阶段 |
| `STATE_LEFT_TURNING` | `4` | 左转执行阶段 |
| `STATE_CROSSWALK` | `5` | 人行道停车阶段 |
| `STATE_SLOW_SIGN` | `6` | 检测到减速标志后的预备阶段 |
| `STATE_SLOWING` | `7` | 减速持续阶段 |
| `STATE_RAMP_ENTER` | `8` | 匝道进入阶段 |
| `STATE_RAMP_EXIT` | `9` | 匝道驶出阶段 |
| `STATE_RIGHT_SIGN` | `10` | 检测到右转标志后的等待阶段 |
| `STATE_RIGHT_TURNING` | `11` | 右转执行阶段 |
| `STATE_NO_ENTRY_SIGN` | `12` | 检测到禁止通行后的预备阶段 |
| `STATE_NO_ENTRY_LEFT_TURNING` | `13` | 禁止通行后的左转执行阶段 |
| `STATE_PEOPLE` | `14` | 行人停车阶段 |

## 5. 编码到动作的详细映射

## 5.1 `sign = 5` 红灯

### 触发条件

- 当前处于 `STATE_NORMAL`
- 识别到 `sign == '5'`

### 状态切换

```text
STATE_NORMAL -> STATE_RED_LIGHT
```

### 当前动作

- 立即输出：
  - `linear.x = 0.0`
  - `angular.z = 0.0`

### 后续恢复条件

在 `STATE_RED_LIGHT` 中：

- 如果识别到 `sign == '3'`
- 切换到 `STATE_GREEN_LIGHT`
- 再从 `STATE_GREEN_LIGHT` 回到 `STATE_NORMAL`

### 迁移要求

- `5` 与 `3` 必须作为成对的红绿灯语义保留
- 后续如果类别体系变化，必须保留“停车信号”与“恢复信号”两种独立语义

## 5.2 `sign = 3` 绿灯恢复

### 当前语义

当前代码里，`3` 不是“普通类别”，而是在红灯状态下被解释为“允许恢复”。

### 生效条件

只有在 `STATE_RED_LIGHT` 中才生效。

### 当前动作

- 切换到 `STATE_GREEN_LIGHT`
- 继承当前巡线速度
- 下一拍回到 `STATE_NORMAL`

### 迁移要求

- 后续不能只看单帧类别名，必须保留“在红灯等待上下文中，某类输出可恢复通行”这一语义

## 5.3 `sign = 4` 左转

### 触发条件

- 当前处于 `STATE_NORMAL`
- `leftsign == 0`
- `rightsign == 0`
- 识别到 `sign == '4'`

### 状态切换

```text
STATE_NORMAL -> STATE_LEFT_SIGN -> STATE_LEFT_TURNING -> STATE_NORMAL
```

### 当前动作

1. `STATE_LEFT_SIGN`
   - 先维持巡线
   - 等待 `turnleftdelay = 0.6s`
2. `STATE_LEFT_TURNING`
   - 以固定转向执行左转
   - 当前参数：
     - `linear.x = 0.0`
     - `angular.z = 1.0`
   - 持续 `turnningtime = 1.8s`

### 额外逻辑

存在一次“等待后再次左转”的补转逻辑：

- `wait_left = 11.8s`
- 在一定条件下会再次进入 `STATE_LEFT_SIGN`

### 迁移要求

- 左转类别必须保留
- 不仅要保留“左转”语义，还要保留其触发后的状态保持与执行时长机制

## 5.4 `sign = 100` 右转

### 触发条件

- 当前处于 `STATE_NORMAL`
- `leftsign == 0`
- `rightsign == 0`
- 识别到 `sign == '100'`

### 状态切换

```text
STATE_NORMAL -> STATE_RIGHT_SIGN -> STATE_RIGHT_TURNING -> STATE_NORMAL
```

### 当前动作

1. `STATE_RIGHT_SIGN`
   - 维持巡线
   - 等待 `turnrightdelay = 0.5s`
2. `STATE_RIGHT_TURNING`
   - 当前参数：
     - `linear.x = 0.0`
     - `angular.z = -1.0`
   - 持续 `turnningtime = 1.8s`

### 迁移要求

- 右转类别必须保留
- 右转控制逻辑与左转不同，不能简单共用一个“turn direction”开关而丢失已有时序

## 5.5 `sign = 0` 人行道

### 触发条件

- 当前处于 `STATE_NORMAL`
- `crosswalksign == 0`
- 识别到 `sign == '0'`

### 状态切换

```text
STATE_NORMAL -> STATE_CROSSWALK -> STATE_NORMAL
```

### 当前动作

1. 延迟阶段：
   - 保持巡线
   - `crosswalkdelay = 0.1s`
2. 停车阶段：
   - `linear.x = 0.0`
   - `angular.z = 0.0`
   - 约 4 秒
3. 中间还存在一小段原地转向逻辑：
   - 在某段时间窗口内 `angular.z = 1.5`

### 迁移要求

- 人行道类别必须保留
- 该类别当前对应的是“停车场景”，不是普通提示类别

## 5.6 `sign = 7` 减速

### 触发条件

- 当前处于 `STATE_NORMAL`
- 识别到 `sign == '7'`

### 状态切换

```text
STATE_NORMAL -> STATE_SLOW_SIGN -> STATE_SLOWING -> STATE_NORMAL
```

### 当前动作

`STATE_SLOW_SIGN` 阶段：

1. 先延迟 `slowdelay = 1.4s`
2. 直行一段：
   - 持续 `delay1 = 2.6s`
3. 再右转一段：
   - `angular.z = slowrightturn = -1.0`
   - 持续 `delay2 = 2.2s`
4. 进入减速状态

`STATE_SLOWING` 阶段：

- `linear.x = 0.13`
- `angular.z = msg_old.angular.z * 0.75`
- 持续 `slowtimedelay = 24.0s`

### 迁移要求

- 减速类别必须保留
- 后续如果优化状态机，也必须保留“预备动作 + 长时减速”这两层语义

## 5.7 `sign = 8` 禁止通行

### 触发条件

- 当前处于 `STATE_NORMAL`
- `noentrysign == 0`
- 识别到 `sign == '8'`

### 状态切换

```text
STATE_NORMAL -> STATE_NO_ENTRY_SIGN -> STATE_NO_ENTRY_LEFT_TURNING -> STATE_NORMAL
```

### 当前动作

1. 预备阶段：
   - 保持巡线
   - 延迟 `noentrydelay = 1.0s`
2. 左转执行阶段：
   - `linear.x = 0.0`
   - `angular.z = 2.0`
   - 持续 `noentry_turnningtime = 1.2s`

### 迁移要求

- 禁止通行类别必须保留
- 这是一个强动作类别，不是仅提示减速

## 5.8 `sign = 2` 匝道

### 当前代码现状

代码中存在匝道逻辑：

```text
STATE_NORMAL -> STATE_RAMP_ENTER -> STATE_NORMAL/STATE_RAMP_EXIT
```

并定义了相关参数：

- `rampdelay = 0.5`
- `rampduration = 2.0`
- `ramp_turnright_angular_z_1 = -1`
- `ramp_turnright_angular_z_2 = 0.4`

### 需要特别注意

当前触发条件写法是：

```python
elif sign == '2' and rampsign != 0:
```

这意味着：

- 从字面看，只有 `rampsign` 非 0 时才会进入匝道逻辑
- 但在进入逻辑前 `rampsign` 初值是 `0`
- 这段逻辑是否按预期生效，需要单独实车或日志确认

### 迁移要求

- 匝道类别是否保留，必须结合实车确认
- 在 YOLOv8 迁移前，不能默认认为当前匝道行为已经完全可靠
- 该类别暂定为“需要保留，但需要专项复核”的类别

## 6. 当前未正式启用或注释掉的逻辑

### 6.1 行人逻辑

代码中存在 `STATE_PEOPLE` 与 `handle_people_emerge_state()`，但正常触发分支当前被注释掉了。

现状说明：

- 代码里保留了行人停车动作
- 但在 `handle_normal_state()` 中，基于 `sign == '3'` 的行人触发逻辑被注释

结论：

- 行人相关逻辑当前不应视为“正式生效规则”
- 如果项目组后续希望恢复，需要单独定义类别与触发标准

### 6.2 `sign = 1` 解除减速

在 `handle_slowing_state()` 中存在关于 `sign == '1'` 的“解除减速”逻辑，但整个分支已被注释掉。

结论：

- `1` 目前不应视为当前生效类别
- 如果后续要恢复，需要重新定义业务含义

## 7. 迁移到 YOLOv8 时的类别保留建议

结合当前代码，建议把类别分成三层。

### A 类：必须保留

这些类别是当前规则层稳定运行的核心输入：

- 红灯
- 绿灯恢复
- 左转
- 右转
- 人行道
- 减速
- 禁止通行

### B 类：建议保留，但必须复核

- 匝道

原因：

- 代码中有逻辑
- 但当前触发条件可能存在实现问题

### C 类：暂不作为当前迁移强制项

- 行人
- 解除减速

原因：

- 当前并非正式启用逻辑
- 不适合作为第一阶段 YOLOv8 迁移强制范围

## 8. 对 YOLOv8 适配层的直接要求

基于当前规则层现状，YOLOv8 适配层至少必须做到：

1. 能输出与当前 `sign` 语义兼容的类别编码
2. 能保证输出稳定，不因单帧抖动频繁切状态
3. 能保留红灯/绿灯、左右转、减速、人行道、禁止通行这些核心类别
4. 能兼容当前 `internal_memory.txt` 接口，作为第一阶段过渡方案

不要求第一阶段就做到：

- 一步替换全部状态机
- 一步废除旧编码
- 一步引入完整新 ROS 接口

## 9. 结论

当前 `velpub.py` 的规则层，真正依赖的不是“完整检测框信息”，而是：

```text
稳定的标志语义编码 + 正确的时序触发
```

因此，YOLOv8 迁移的第一性任务不是“让模型输出更多信息”，而是：

**先稳定复现当前规则层真正依赖的那几个关键类别与行为触发条件。**

这就是后续 YOLOv8 迁移设计、适配器设计和接口设计的第一份基线文档。
