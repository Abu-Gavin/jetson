# YOLOv8 迁移总纲（小车现状版）

## 1. 文档定位

本文不是详细实施清单，也不是代码设计细节文档，而是面向本车当前状态的“上位总纲”。

它回答 4 个核心问题：

1. 为什么现在要迁移到 YOLOv8
2. 这台小车当前到底受什么约束
3. 后续改造必须遵守什么边界
4. 代码改造应该按什么顺序推进

后续所有代码改动、接口设计、部署方案和联调步骤，都应以本文为约束。

## 2. 当前小车的客观现状

### 2.1 当前真正能跑车的主链

这台车当前真正能稳定跑起来的核心链路不是“YOLO 控车”，而是：

```text
detect_line3.py
-> note.txt
-> vth2ros.py
-> /cmd_vel 或 /cmd_vel_1
-> velpub.py（可选仲裁）
-> /cmd_vel
-> wheeltec_robot_node
-> 底盘
```

换句话说：

- 巡线链是当前主能力
- YOLO 链当前属于规则仲裁输入，不是整车唯一主控制来源

### 2.2 当前 YOLO 在系统里的真实位置

当前 YOLO 的作用不是直接发布标准 ROS 检测消息供主链消费，而是：

```text
DeepStream-Yolo
-> internal_memory.txt
-> velpub.py
-> 交通规则状态机
```

所以当前系统不是“换一个模型就结束”，而是“模型、部署方式、输出接口、规则仲裁”是耦合在一起的。

### 2.3 当前旧 YOLO5 方案的根本问题

当前旧方案的根本问题不在“YOLOv5 精度是否够用”，而在：

1. 部署链老化
2. 输出接口非标准
3. 与新环境兼容性差
4. 维护成本高

具体表现为：

- 旧 `YOLOv5 cfg/wts` 路线在 `DeepStream 7.1 / TensorRT 10` 下不稳定
- `internal_memory.txt` 是文件接口，缺少类型约束、版本约束和消息可观测性
- 规则层 `velpub.py` 依赖硬编码标志编码，迁移成本集中在“输出兼容”

### 2.4 当前不能破坏的能力

在 YOLOv8 迁移过程中，以下能力必须视为“不可先破坏”的底线能力：

1. 小车底盘基础驱动必须可用
2. 本地巡线链必须可独立运行
3. Windows 侧网页视频流能力必须保留
4. 交通规则识别链即便失败，也不能拖垮整车主链

这意味着后续改造必须坚持：

- 巡线链和识别链解耦
- 新识别链先旁路运行
- 新识别链稳定后再进入仲裁

## 3. 本项目的真实迁移目标

本项目长期目标不是“把 YOLO5 换成 YOLO8”这么简单，而是 4 件事一起完成：

1. 模型基座从 YOLOv5 转到 YOLOv8
2. 部署链从旧式 `cfg/wts` 路线切到现代 ONNX / TensorRT / 新 DeepStream 路线
3. 输出接口从 `internal_memory.txt` 逐步升级到标准化接口
4. 规则仲裁层从“依赖历史编码”演进到“依赖明确消息契约”

如果只做第 1 件，不做后 3 件，项目仍然会长期背负历史包袱。

## 4. 本车专用的迁移原则

后续所有改造必须遵守以下原则。

### 原则 A：先保巡线，再改识别

这台车当前真正能稳住车辆行为的是巡线链，不是 YOLO 链。

所以：

- 识别链升级不能反向破坏巡线链
- 任何时候都要能退回“纯巡线可跑”状态

### 原则 B：先并行验证，再接入控制

YOLOv8 必须先在 Jetson 上独立运行、独立验证、独立出结果。

禁止一上来就：

- 直接替换现有 YOLO 输入
- 直接改 `velpub.py` 让整车依赖新链
- 直接把新检测结果接到底盘控制

### 原则 C：先兼容旧规则层，再重构旧规则层

这台车的历史负担主要在 `velpub.py`。

因此最稳的顺序是：

1. 先让 YOLOv8 输出兼容旧规则层
2. 再让旧规则层跑在新检测输入上
3. 最后再重构旧规则层本身

不要一开始同时改：

- 模型
- 推理链
- 输出接口
- 状态机
- launch

这样风险太大。

### 原则 D：先双轨，后切主

迁移期必须同时存在：

- 旧链可回退方案
- 新链验证方案

新链只有在满足阶段验收标准后，才能进入“主链接管”阶段。

### 原则 E：接口升级优先于逻辑升级

当前系统最大的长期技术债，不是某一个 if/else，而是接口方式。

所以长期最重要的不是“先优化某个转弯状态机”，而是：

- 先把检测输出接口稳定下来
- 先把消息契约定清楚
- 再在稳定接口上做逻辑演进

## 5. 总体技术路线

针对这台车，推荐的长期技术路线如下：

```text
YOLOv8
-> ONNX
-> TensorRT / DeepStream 7.1 新方案
-> 检测适配层
-> 旧规则层兼容
-> 标准 ROS 接口
-> 新规则层演进
```

这里最关键的是“检测适配层”。

检测适配层的职责是：

- 接收 YOLOv8 原始检测结果
- 做类别映射、连续确认、稳定性过滤
- 对外兼容旧规则输入
- 同时为未来标准 ROS 接口留出口

这个适配层必须被视为独立模块，而不是散落在 `velpub.py` 里临时拼接。

## 6. 改造顺序总纲

后续代码改造必须按以下总顺序推进。

## 第一层：冻结现状

在开始任何大改前，先冻结以下内容：

1. 当前规则类别与行为映射
2. 当前 `velpub.py` 真正依赖的输入字段
3. 当前 launch 与实际启动顺序
4. 当前可跑主链与回退方式

这一层不以“改代码”为目标，而以“锁边界”为目标。

## 第二层：独立落地 YOLOv8

先把 YOLOv8 作为一个独立检测能力跑起来。

要求：

- 不接管小车控制
- 不依赖旧 `velpub.py`
- 可以离线图像、离线视频、实时摄像头独立验证

此阶段只证明：

- 新模型可跑
- 推理速度可接受
- 目标类别能识别

## 第三层：建立适配层

在 YOLOv8 之上增加“检测适配层”，输出兼容当前规则层的结果。

短期可兼容：

- `internal_memory.txt`

中期必须支持：

- 标准 ROS topic / 标准 message

这个阶段的核心，不是“检测准不准”，而是“输出能不能稳定、可控、可调”。

## 第四层：让新检测驱动旧规则层

当适配层稳定后，再让 `velpub.py` 使用新检测结果。

此阶段：

- 仍然尽量少动旧状态机主干
- 先验证行为一致性
- 重点观察误触发、漏触发、持续时间和状态切换时机

## 第五层：整车切主

只有在下列条件同时满足后，新链才能切为主链：

1. 独立检测稳定
2. 适配层稳定
3. 旧规则层已能被新检测驱动
4. 实车关键场景通过
5. 具备快速回退能力

## 第六层：正式去历史包袱

当新链已稳定接管后，再做以下动作：

1. 逐步淘汰 `internal_memory.txt`
2. 逐步淘汰旧 YOLO5 部署相关路径
3. 逐步把旧规则层输入改成标准 ROS message
4. 视情况重构 `velpub.py` 的全局变量与状态机实现

## 7. 本项目中什么先不改

为了降低风险，以下内容在前期不应优先大改：

1. `detect_line3.py` 巡线核心逻辑
2. `vth2ros.py` 的 PID 主逻辑
3. `wheeltec_robot_node` 底盘驱动主逻辑
4. 当前网页视频流机制
5. 整车启动链的底层基础依赖

原因很简单：

- 这些部分当前已经是“可用基础能力”
- 它们不是本轮 YOLO 升级的主要矛盾
- 过早同时修改会显著增加联调复杂度

## 8. 本项目中什么必须先改

以下内容必须优先进入改造范围：

1. YOLO 输出接口定义
2. YOLO 类别到规则编码的映射
3. 新检测链独立部署方式
4. 检测适配层
5. `velpub.py` 的输入来源抽象

如果这些部分不先动，后面所有联调都会变成临时 patch。

## 9. 阶段性验收口径

为了避免“改了很多但无法判断是不是成功”，本项目必须按阶段验收。

### 阶段验收 1：检测可用

标准：

- YOLOv8 在 Jetson 上稳定运行
- 指定类别可识别
- 速度满足最低要求

### 阶段验收 2：输出可兼容

标准：

- 新检测输出可被适配成旧规则层输入
- 输出连续稳定，不大幅抖动

### 阶段验收 3：规则可接管

标准：

- `velpub.py` 可被新链驱动
- 核心规则行为可复现

### 阶段验收 4：整车可运行

标准：

- 巡线 + 规则仲裁可同时稳定运行
- 关键场景实车通过
- 异常场景可降级

### 阶段验收 5：旧接口可下线

标准：

- `internal_memory.txt` 不再是唯一依赖
- 系统可完全通过标准接口运行

## 10. 项目组协作总纲

建议项目组后续按以下 4 条线并行协作：

### 线 1：模型线

负责：

- YOLOv8 训练
- 类别定义
- 数据集和精度验证

### 线 2：部署线

负责：

- ONNX / TensorRT / DeepStream 方案
- Jetson 实机推理
- 性能测试

### 线 3：接口线

负责：

- 检测适配层
- 消息契约
- 新旧接口兼容

### 线 4：整车线

负责：

- `velpub.py` 对接
- 实车联调
- 行为验证
- 回退机制

## 11. 总纲结论

针对这台小车，YOLOv8 迁移的正确打开方式不是：

```text
直接替换模型 -> 直接接管控制
```

而应该是：

```text
冻结现状
-> 独立落地 YOLOv8
-> 建适配层
-> 驱动旧规则层
-> 小范围接管
-> 整车切主
-> 下线历史接口
```

因此，后续代码改造的总要求可以概括成一句话：

**先把 YOLOv8 做成“稳定的新检测能力”，再把它做成“可接管旧规则链的输入能力”，最后才把它做成“新的整车默认识别主链”。**

这就是本车后续所有 YOLOv8 改造的总纲。
