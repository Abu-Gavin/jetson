# 当前主链运行逻辑总览

## 1. 文档定位

这份文档只解释 `yahboom_ws` 里当前真正用于跑车的主链，不把 `wheeltec_yolo_action`、`camera`、`largemodel` 这些非当前启动链的包当成主流程。

当前重点对应的启动命令是：

```bash
ros2 launch turn_on_wheeltec_robot turn_on_wheeltec_robot.launch.py
ros2 launch velpub detect_line.launch.py
ros2 launch velpub AIcar_move.launch.py
ros2 launch velpub get_yolo_rst.launch.py
```

需要先说明一个关键事实：

- 按当前代码，`AIcar_move.launch.py` 默认把 `vth2ros.py` 输出到 `/cmd_vel`
- `get_yolo_rst.launch.py` 启动的 `velpub.py` 默认订阅 `/cmd_vel_1`
- 所以如果 4 条命令完全按上面原样执行，`velpub.py` 会启动，但不会接到 `vth2ros.py` 的巡线输入

也就是说：

- 这 4 条命令原样启动时，真正闭环的是“本地巡线直出底盘”链
- 如果要让 `velpub.py` 参与 YOLO/交通规则仲裁，`AIcar_move.launch.py` 需要改成把输出发到 `/cmd_vel_1`

## 2. 启动命令与实际节点

| 启动命令 | 实际启动内容 | 当前默认关键参数 | 在主链里的角色 |
| --- | --- | --- | --- |
| `ros2 launch turn_on_wheeltec_robot turn_on_wheeltec_robot.launch.py` | `wheeltec_robot_node` | 串口 `/dev/ttyUSB0`，波特率 `115200`，launch 默认 `odom_frame_id=odom_combined` | 底盘执行层，最终订阅 `/cmd_vel` 并通过串口下发 |
| `ros2 launch velpub detect_line.launch.py` | `detect_line3.py`，节点名 `detect_line` | `video_device=/dev/video3`，`publish_cmd_vel=false`，`note_file=<install-prefix>/lib/velpub/note.txt` | 本地巡线感知层，产出 `deviation,severity` |
| `ros2 launch velpub AIcar_move.launch.py` | `vth2ros.py`，节点名 `vth2ros` | 默认 `cmd_vel_topic=/cmd_vel` | 把 `note.txt` 转成巡线速度指令 |
| `ros2 launch velpub get_yolo_rst.launch.py` | `velpub.py`，节点名 `velpub` | 默认 `input_topic=/cmd_vel_1`，`output_topic=/cmd_vel` | 规则仲裁层，读取 YOLO 结果并覆盖或修正速度 |

## 3. 两条必须区分的控制闭环

### 3.1 按这 4 条命令原样启动时

当前代码下，真实闭环是：

```text
摄像头帧
-> detect_line3.py
-> note.txt + /lane/steering_angle + /lane/turn_severity
-> vth2ros.py
-> /cmd_vel
-> wheeltec_robot_node
-> 串口底盘
```

此时：

- `detect_line3.py` 负责本地采图、本地 UNet 推理、本地网页流
- `vth2ros.py` 负责 PID 和弯道速度/转向调节
- `wheeltec_robot_node` 负责真正下发底盘
- `velpub.py` 虽然已启动，但它在等 `/cmd_vel_1`
- 如果没有别的节点往 `/cmd_vel_1` 发巡线速度，`velpub.py` 不会成为主控制链的一环

### 3.2 想启用 YOLO/交通规则仲裁时

如果目标是让 `velpub.py` 真正参与控制链，应使用：

```bash
ros2 launch turn_on_wheeltec_robot turn_on_wheeltec_robot.launch.py
ros2 launch velpub detect_line.launch.py
ros2 launch velpub AIcar_move.launch.py cmd_vel_topic:=/cmd_vel_1
ros2 launch velpub get_yolo_rst.launch.py
```

这时真实闭环才是：

```text
摄像头帧
-> detect_line3.py
-> note.txt
-> vth2ros.py
-> /cmd_vel_1
-> velpub.py + internal_memory.txt
-> /cmd_vel
-> wheeltec_robot_node
-> 串口底盘
```

这条链里：

- `/cmd_vel_1` 是“巡线原始输出”
- `/cmd_vel` 是“仲裁后的最终输出”
- `velpub.py` 通过读取 DeepStream-Yolo 写出的 `internal_memory.txt` 决定是否停车、左转、右转、减速、过人行道或处理匝道

## 4. 主链节点职责

### 4.1 `turn_on_wheeltec_robot` / `wheeltec_robot_node`

职责：

- 订阅 `/cmd_vel`
- 把 `linear.x`、`linear.y`、`angular.z` 转成串口控制帧
- 周期查询底盘状态
- 发布 `/odom`、`/imu1`、`/PowerVoltage`

结论：

- 它是整条链的最终执行端
- 不关心控制来自巡线、仲裁还是其他节点，只认 `/cmd_vel`

### 4.2 `velpub/detect_line.launch.py` -> `detect_line3.py`

职责：

- 打开本地摄像头
- 加载 `lane_local_core.py` 中的 UNet 与 `LaneTracker`
- 对每帧图像计算 `deviation,severity`
- 把结果写入 `note.txt`
- 发布 `/lane/steering_angle` 和 `/lane/turn_severity`
- 在 `8090` 端口提供 `Original`、`Mask`、`Fit` 三路网页流

关键点：

- 当前 launch 已固定启动 `detect_line3.py`，不是 `detect_line2.py`
- 当前 launch 默认 `publish_cmd_vel=false`，所以它默认不直接给底盘发速度

### 4.3 `velpub/AIcar_move.launch.py` -> `vth2ros.py`

职责：

- 读取 `note.txt`
- 取出 `deviation,severity`
- 用 PID 计算转向
- 用 `severity` 做弯道降速和转向增强
- 发布巡线速度指令

关键点：

- 节点内部默认 `cmd_vel_topic=/cmd_vel_1`
- 但当前 launch 文件默认把它覆盖成了 `/cmd_vel`
- 所以看 launch 行为时，要以 launch 默认值为准，不要只看脚本内部默认值

### 4.4 `velpub/get_yolo_rst.launch.py` -> `velpub.py`

职责：

- 订阅巡线原始速度
- 读取 `/opt/nvidia/deepstream/deepstream-6.0/sources/DeepStream-Yolo/nvdsinfer_custom_impl_Yolo/internal_memory.txt`
- 依据交通标志状态机决定：
  - 正常放行
  - 红灯停车 / 绿灯恢复
  - 左转 / 右转
  - 人行道停车
  - 减速区减速
  - 匝道动作
  - 禁止通行改道
- 将最终速度发布到 `/cmd_vel`

当前代码里可直接看到的主要标志编码包括：

- `0`：人行道
- `2`：匝道
- `4`：左转
- `5`：红灯
- `7`：减速
- `8`：禁止通行
- `100`：右转
- `3`：在红绿灯状态机中被当作绿灯恢复

## 5. 关键接口与文件契约

### 5.1 ROS 话题

- `/lane/steering_angle`
  - `detect_line3.py` 发布的横向偏差
- `/lane/turn_severity`
  - `detect_line3.py` 发布的弯道强度
- `/cmd_vel_1`
  - 预留给 `vth2ros.py` 输出巡线原始速度，供 `velpub.py` 订阅
- `/cmd_vel`
  - 最终给底盘的速度指令
- `/odom`、`/imu1`、`/PowerVoltage`
  - 底盘反馈

### 5.2 文件接口

- `note.txt`
  - 默认由 launch 指向 `<install-prefix>/lib/velpub/note.txt`
  - 内容格式为 `deviation,severity`
  - 上游写入者：`detect_line3.py`
  - 下游读取者：`vth2ros.py`
- `internal_memory.txt`
  - 当前代码硬编码路径为 `/opt/nvidia/deepstream/deepstream-6.0/sources/DeepStream-Yolo/nvdsinfer_custom_impl_Yolo/internal_memory.txt`
  - 上游写入者：DeepStream-Yolo 自定义插件
  - 下游读取者：`velpub.py`

### 5.3 底盘串口

- 默认设备：`/dev/ttyUSB0`
- 默认波特率：`115200`
- 真正把 `/cmd_vel` 送到硬件的是 `wheeltec_robot_node`

## 6. 当前主链之外的包

### 6.1 `wheeltec_yolo_action`

这是另一套旧视觉控制链，不属于当前这 4 条命令的主流程。

它的特点是：

- 使用自己的 `dp_drive.launch.py` / `full_stack.launch.py`
- 依赖 `/usb_cam/image_raw`、`/darknet_ros/bounding_boxes`、`/object_tracker/current_position`
- 通过 `drive_decision_node.py` 和 `drive_execution_node.py` 配合产出 `/cmd_vel`

结论：

- 这是“另一套可独立运行的视觉控制链”
- 不是当前 `velpub + detect_line3 + vth2ros + velpub.py` 这一套主链

### 6.2 `camera`

当前目录下多数文件为空文件或占位文件，不是当前主链依赖。

### 6.3 `largemodel`

当前目录下大量 Python、launch、yaml 文件为空文件或占位内容，不是当前开车主链依赖。

### 6.4 `text_chat`

这是附加的人机文本交互模块，不影响车辆主控制闭环。

所以它适合做“查接口”，不适合单独拿来理解当前整车实际怎么跑。

## 8. 验收结论

如果只问当前主链最关键的 4 个问题，答案如下：

- 为什么要按这几条命令启动：
  - `turn_on_wheeltec_robot` 负责底盘
  - `detect_line` 负责本地巡线感知
  - `AIcar_move` 负责把巡线误差变成速度
  - `get_yolo_rst` 负责把 YOLO 交通规则接入速度仲裁
- 每条命令启动了谁：
  - `wheeltec_robot_node`
  - `detect_line3.py`
  - `vth2ros.py`
  - `velpub.py`
- 最终谁在发 `/cmd_vel`：
  - 原样 4 条命令时，主要是 `vth2ros.py`
  - 启用 `/cmd_vel_1 -> velpub.py -> /cmd_vel` 仲裁链时，最终是 `velpub.py`
- YOLO 在当前链里通过什么介入控制：
  - 不是直接发 ROS 检测框进这条主链
  - 而是通过 `internal_memory.txt` 被 `velpub.py` 读取并进入状态机
